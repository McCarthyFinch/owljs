obol

Status

  • experimental

When completed, will replace the old obol code for generating equivalence axioms from strings

Rules

{
 head: "NAME",
 body: TOKENLIST,
 gfun: FUNCTION
}

Each token in a token list is either a terminal (a string) or a non-terminal.  Currently the only non-terminals allowed are variables, objects keyed by minimally ‘name’, and optionally ‘type.

Example

{
    head: "cell differentiation",
    body: [{name: "cell", type: "cell"}, " differentiation"],
    gfun: function(o, h) {
        return intersectionOf(
            o.cell_differentiation,
            someValuesFrom(o.results_in_acquisition_of_features_of, h.cell));

    },
    score : 15
}

This matches any class whose label, synonym, etc ends with “differentiation”.  The start of the string must correspond to the label of a cell type.  Each match generates a class expression.

See the directory owljs/obol/ for a list of ready-made parse rules for bio-ontologies

useOboVocab

Obol.prototype.useOboVocab = function()

Set this if the default OBO Vocabulary and scores are to be used

  • labels and exact synonyms score high
  • broad, narrow and related synonyms are lower

This also includes the NIF “obo” annotation property for synonym TODO IAO

restrictProperties

Obol.prototype.restrictProperties = function(ps)

restrict the set of annotation properties used to find a lexical label for a term to the specified set.

if no annotation vocabulary has been set, the specified set of properties will be used with a default score.

A typical scenario is to call useOboVocab(), and then to restrict the set (e.g. to exact syns and labels) using this method

Arguments

  • ps : owlapi.OWLAnnotationProperty [] (or a list of IRIs)
